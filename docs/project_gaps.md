# 法律案例 RAG 检索问答系统项目不足与优化方向

## 当前不足

1. 前端仍是 FastAPI 内嵌 HTML，适合演示，但不适合长期维护；后续应拆成 React/Vue 项目并组件化案卷、会话、证据面板。
2. 当前没有登录、权限和租户隔离，默认单用户模式只能用于本地演示，不能直接开放给多人使用。
3. 案件记忆已有 LLM 抽取和规则 fallback，但还缺少字段置信度、冲突事实处理和人工确认机制。
4. 多轮上下文目前采用最近轮次 + 长度截断，尚未做 token 级预算、重要事实排序和长期记忆召回。
5. RAG 已支持向量检索、关键词检索、RRF 融合和可选 reranker，但还缺少召回率、引用命中率等离线评测来证明效果。
6. 完整导入 12 万条数据耗时较长，面试演示通常使用 `INGEST_LIMIT` 小规模导入；后续可补充批量导入进度、断点续跑和索引调优。
7. DeepSeek 云端模型依赖环境变量配置，缺少密钥检查页面、额度提示和请求失败后的前端可解释反馈。
8. 已有 `/chat/stream` 流式输出接口，但前端仍是简单 SSE 展示，缺少取消生成、重试和更细粒度的状态反馈。
9. 文件上传、OCR、合同解析仍未接入真实后端流程，前端不应夸大该能力。
10. 法律安全边界还需加强，包括免责声明、不可替代律师意见、敏感信息脱敏和高风险场景拒答策略。
11. 测评集规模较小，缺少可量化指标，如召回率、引用命中率、拒答准确率和幻觉率。
12. 部分历史文件曾出现中文乱码，后续应统一 UTF-8 编码，并逐步清理旧测试和文档中的乱码文本。

## 推荐优化优先级

1. 产品可用性：拆分前端项目，增加登录、案卷权限、错误提示、生成取消和重试能力。
2. 上下文系统：引入 token 预算、案件事实结构化抽取、长期记忆检索和重要事实置顶。
3. 检索质量：补充召回评测集、索引参数调优、reranker 性能评估和引用覆盖率检查。
4. 法律可信度：完善拒答策略、风险提示、补证建议和引用覆盖率检查。
5. 工程交付：补充 Docker 部署、CI 测试、日志追踪、配置校验和 README 故障排查。

## 简历表达建议

当前版本已经可以写成“完成法律 RAG 检索问答系统的工程化升级”。但如果要投大模型应用开发岗位，最值得继续补强的是：

- 多轮上下文与案件记忆系统的设计细节。
- RAG 检索质量评测与引用可信度控制。
- DeepSeek 云端模型接入、fallback 和异常处理。
- PostgreSQL 持久化会话、案卷和记忆的产品化能力。
